

# 社交App“**岸**”

# 技术与数据”板块全系列细分文档清单

### 第一板块：架构与设计
这部分文档回答的是“这个App是怎么盖起来的”。

1.  **系统架构图**：一张高层次的图，展示App客户端、后端服务器、数据库、缓存、对象存储之间的交互关系。比如用户发一条匿名消息，请求是先到负载均衡，再到应用服务器，最后写入数据库。
2.  **技术选型文档**：记录为什么选择某个技术。例如：为什么用 React Native 开发跨平台App？为什么数据库用 MySQL 而不是 PostgreSQL？为什么服务器放在阿里云而不是腾讯云？
3.  **数据库设计文档**：核心资产。包含**ER图**，以及每一张表的字段说明。例如“用户匿名信息表”包含哪些字段（匿名ID、最后登录时间、注册时间、设备指纹等），“匿名消息表”包含哪些字段（消息ID、发送方ID、接收方ID、内容、发送时间、是否已读）。
4.  **接口文档**：规定App和服务器之间怎么“说话”。包含每一个接口的URL、请求方式、请求参数、返回示例。例如“发布一条匿名动态”的接口，需要传什么参数，返回成功或失败的格式是什么。
5.  **数据结构定义**：如果App有复杂的本地数据（比如草稿箱、缓存），需要定义数据在手机本地是怎么存的。

### 第二板块：开发与规范
这部分确保代码质量和团队协作。

6.  **代码规范**：强制或推荐的编码风格。例如 Java 代码变量命名用驼峰、缩进用4个空格、Git 提交信息格式为 `[模块] 改动描述`。
7.  **Git 分支管理流程**：团队怎么协作开发。例如 `master` 分支永远是稳定版，`develop` 是开发版，`feature/xxx` 用来开发新功能，`hotfix/xxx` 用来紧急修 Bug。
8.  **代码 Review 清单**：开发人员提交代码审查前需要自检的列表。包含“代码有没有注释”、“有没有处理异常”、“有没有写单元测试”、“有没有安全漏洞”。
9.  **第三方服务集成文档**：“岸”可能用到很多第三方服务，如极光推送（发通知）、融云/环信（做聊天）、友盟（做统计）、阿里云OSS（存图片）。记录下每个服务的接入方式、配置参数、注意事项。

### 第三板块：测试与质量
这部分保证产品上线后不出大问题。

10. **测试计划**：大版本发布前的测试安排。包含测试范围、测试环境、测试时间、人员安排、通过标准。
11. **测试用例**：测试人员用来一条条执行的手册。例如：
    -   **功能测试**：匿名发一条纯文本消息 → 预期：对方能收到。
    -   **异常测试**：发消息时断网 → 预期：App 提示“网络开小差了”，恢复网络后能自动重发。
    -   **兼容性测试**：在 iPhone 12 和 iPhone 14 上分别测试，界面是否错位。
12. **自动化测试脚本**：如果有自动化测试，对应的代码和说明文档。
13. **性能测试报告**：例如“2000人同时在线发消息时，服务器响应时间从 50ms 上升到了 200ms，CPU 占用率 70%”，记录压测结果和优化建议。
14. **Bug 管理规范**：记录发现 Bug 后怎么提、怎么分优先级。P0 级 Bug（核心功能不能用）必须立即修，P3 级 Bug（界面错位）可以下个版本修。

### 第四板块：部署与运维
这部分保证App能稳定运行，不宕机。

15. **部署文档**：如何把代码变成能访问的服务。包含环境要求、配置文件说明、部署步骤、启动停止脚本。
16. **服务器清单**：Excel 或在线表格，记录所有服务器的 IP、用途、配置、所属项目、负责人、过期时间。
17. **监控与告警配置**：监控什么指标。例如“服务器 CPU 超过 80% 告警”、“接口 5xx 错误率超过 1% 告警”、“匿名消息发送量突然下跌 50% 告警”。包含告警阈值、接收人、处理方式。
18. **灾备与恢复手册**：万一数据丢了怎么办。包含数据库备份策略（每天全量备份+每半小时增量备份）、备份文件存储位置、恢复步骤。
19. **故障应急响应手册**：线上出大事时的标准操作流程。例如：
    -   **第一步**：确认故障，拉群，发公告。
    -   **第二步**：根据现象判断是数据库问题还是代码问题。
    -   **第三步**：执行回滚或限流。
    -   **第四步**：故障复盘。

### 第五板块：数据与指标
这部分回答“我们的产品运营得怎么样”。

20. **埋点文档**：记录了在App的哪些地方埋了点，收集什么数据。例如在“发布匿名动态”按钮上埋点，记录点击次数、发布成功率、发布耗时。这是数据分析的基础。
21. **核心数据定义**：统一公司内部对指标的口径。例如“日活用户”定义为当天打开过App且前台运行超过1秒的去重设备数；“用户留存率”定义为第N天再次打开App的用户比例。
22. **数据报表需求文档**：给数据开发人员看的，说明需要哪些报表。例如：
    -   **每日数据看板**：日活、新增、留存、平均使用时长。
    -   **匿名聊天监控看板**：总消息量、匹配成功对数、举报率。
23. **数据分析报告**：周期性产出。例如“春节期间用户活跃度下降了20%，可能是因为大家忙着聚会，没空刷手机”，或者“新版本上线后，举报率上升了5%，怀疑是推荐算法出了问题”。

### 第六板块：安全与隐私
匿名社交App尤其重视这一块。

24. **安全设计文档**：记录系统做了哪些安全防护。例如用户密码是经过加盐哈希存储的、匿名 ID 是随机生成的无法反查手机号、全站启用 HTTPS、针对 SQL 注入和 XSS 的防护措施。
25. **隐私数据清单**：记录产品收集了哪些用户隐私数据。例如设备信息、大致地理位置、聊天记录。这份文档主要用于应对隐私合规审查。
26. **数据脱敏规范**：在测试环境或数据分析时，如何处理敏感信息。例如导出数据库时，把手机号中间四位变成 ****。
27. **安全审计报告**：定期请安全团队或第三方扫描系统漏洞后生成的报告，以及修复情况。

这些文档共同构成了“岸”的技术底座。从代码规范到服务器监控，从埋点数据到安全防护，每一份文档都在支撑着这款匿名社交App的稳定、安全和持续迭代。
